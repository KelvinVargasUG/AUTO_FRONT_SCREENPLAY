# Design: Front-End Automation with Screenplay

## Technical Approach

Implement the acceptance criteria for successful/failed user registration and successful invoice generation using the Screenplay Pattern with Serenity BDD in Java. This decouples the what, how, and why of automation, establishing clear responsibilities among Actors, Tasks, Questions, and Targets.

## Architecture Decisions

### Decision: Screenplay Pattern over Page Factory / POM
**Choice**: Use Serenity BDD's Screenplay pattern.
**Alternatives considered**: Traditional Page Object Model (POM) or Page Factory.
**Rationale**: Screenplay aligns perfectly with the goal of separating intent from implementation, making steps highly reusable and scenarios robust against UI changes. Required by specification (PDR).

### Decision: Direct injection of Serenity core components
**Choice**: Use Serenity's built-in environment variable handling and dependency injection rather than custom wrappers.
**Alternatives considered**: Custom `EnvironmentVariablesUtils` or `@Autowired` Spring configurations.
**Rationale**: Avoid boilerplate. Serenity natively injects `EnvironmentVariables` into `@Step` components. (Learned from previous migrations like AUTO_FRONT_POM_FACTORY).

### Decision: Dynamic Selectors handled inside UI classes
**Choice**: Use factory methods to create dynamic Targets (e.g. `Target.the("table {0}").locatedBy("div[data-testid='table-item-{0}']")`).
**Alternatives considered**: Hardcoded XPath/CSS selectors in tasks.
**Rationale**: Keeps Tasks clean and confines localized CSS/XPath knowledge strictly to UI classes.

## Data Flow

Data flows primarily from the StepDefinitions to the Tasks, which interact with UI Targets. Questions query the UI state and report back to StepDefinitions.

    Feature File (Gherkin)
         │
    StepDefinition (Actor)
         │
    Task (Performs actions using Data Models) => [UserData, OrderData]
         │
    Target / UI (Locators)
         │
    Question (Asks for state) ──→ StepDefinition (Asserts Answer)

| File | Action | Description |
|------|--------|-------------|

## Interfaces / Contracts

```java
// Example UserData model structure
public record UserData(
    String name,
    String email,
    String password
) {}

// Example parameterized Target in UI class
public static Target TABLE_ITEM(String tableId) {
    return Target.the("Tabla " + tableId).locatedBy("div[data-testid='table-item-{0}']").of(tableId);
}
```

## Testing Strategy

| Layer | What to Test | Approach |
|-------|-------------|----------|
| Unit | Utility methods | Basic JUnit 5 assertions for formatting or data generation utils, if any. |
| Integration | Tasks | Covered primarily by E2E since tasks directly bind to UI state in Screenplay. |
| E2E | Core flows (Auth, Orders) | Full execution using Serenity Screenplay + Cucumber. The actor performs steps locally via WebDriver. |

## Migration / Rollout

No migration required. This is a greenfield automation project. Note that `AppTest.java` that is auto-generated by gradle scaffolding should be deleted to prevent runner pollution.
